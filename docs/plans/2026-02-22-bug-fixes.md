## Bug 2: Can't delete published marketplace templates (Medium)

**Files:** `convex/templates.ts:remove`, `convex/marketplace.ts:unpublish`

**Problem:** `templates.remove` deletes the template row but doesn't clean up:
- The `marketplace` entry remains (orphaned)
- `marketplaceBlocks` remain (orphaned)
- User has no UI path to delete a marketplace entry directly

**Fix:** In `templates.remove`, before deleting, check if `publishedMarketplaceId` exists. If so, call the same cleanup logic as `unpublish`:
1. Delete associated `marketplaceBlocks`
2. Delete the `marketplace` entry
3. Then delete the template

```ts
// In templates.remove handler, before ctx.db.delete(args.id):
const template = await ctx.db.get(args.id)
if (template?.publishedMarketplaceId) {
  const mpId = template.publishedMarketplaceId
  // Delete marketplace blocks
  const mpBlocks = await ctx.db
    .query("marketplaceBlocks")
    .withIndex("by_marketplace", (q) => q.eq("marketplaceId", mpId))
    .collect()
  for (const b of mpBlocks) {
    await ctx.db.delete(b._id)
  }
  // Delete marketplace entry
  const mpEntry = await ctx.db.get(mpId)
  if (mpEntry) await ctx.db.delete(mpId)
}
await ctx.db.delete(args.id)
```

## Bug 3: Brainstorm retry sends stale history for Claude (Medium)

**File:** `src/hooks/useBrainstorm.ts`

**Problem:** `sendMessageClaude` (line 437) takes only `content: string` and rebuilds conversation history from its closed-over `messages` state. When called from `retryMessage` or `editMessage`, the closure captures the pre-truncation/pre-edit messages — so the backend receives the full old history including messages the user wanted to retry/replace.

`sendMessageOllama` and `sendMessageOpenRouter` both accept a `conversationHistory` parameter and work correctly.

**Fix:** Add `conversationHistory` parameter to `sendMessageClaude`, matching the other two providers:

1. Change signature: `async (content: string, conversationHistory: { role: "user" | "assistant"; content: string }[])`
2. Remove the internal `messages.map(...)` that rebuilds history
3. Use the passed-in `conversationHistory` directly
4. Update all call sites (`sendMessage`, `retryMessage`, `editMessage`) to pass `conversationHistory` — they already compute it correctly

This also fixes the same bug in `editMessage` (line 660).

## Feature 4: Persist brainstorm conversations in localStorage (Medium)

**File:** `src/hooks/useBrainstorm.ts`

**Problem:** Brainstorm messages are ephemeral `useState`. If tokens run out mid-brainstorm, browser crashes, or user navigates away — everything is lost.

**Fix:** Save/load messages to localStorage keyed by sessionId.

1. Add keys to settings pattern: `contextforge-brainstorm-{sessionId}`
2. On `setMessages` — debounce-save to localStorage (every 2s or on message complete)
3. On hook init (when `sessionId` changes) — load from localStorage if messages exist
4. On `clearConversation` — also clear localStorage entry
5. Cap stored conversations (e.g. last 5 sessions) to avoid unbounded storage growth
6. Add a small "Conversation restored" toast/indicator when loading from storage

**Storage format:**
```ts
{
  messages: Message[],
  savedAt: number,
  provider: string,
  model: string | null
}
```
